1. OBJETIVO
--------------------------------------------------------------------------------
Construir "TIL TRACKER", una app web (nueva, independiente) donde un usuario:
  1. Ingresa una LLAVE y ve ÚNICAMENTE las maniobras cuyo campo man_moni_key
     (colección "maneuver" de maylob_backend) coincide con esa llave.
  2. SEGURIDAD: el usuario NO puede usar el campo de llave (ni ningún otro
     vector) para inyectar código, operadores NoSQL o payloads que extraigan
     datos de la base de datos. Ver sección 6 (modelo de amenazas y defensas).
  3. La llave se guarda en el navegador (localStorage) y NO se vuelve a pedir,
     salvo que la llave ya no exista en la BD o haya sido cambiada por el
     administrador (en ese caso se borra la guardada y se vuelve a solicitar).
  4. VISTA RÁPIDA por maniobra: contenedores, transportista, avance, ubicación
     y estatus + botón desplegable para ver TODOS los datos.
  5. En el detalle: sección con CANVAS para el link de GPS (man_gps_link).
     Sin link registrado -> el canvas muestra la leyenda "MAPA NO DISPONIBLE".
  6. SOLO LECTURA: el usuario no puede editar ningún campo.
  7. Diseño: GLASSMORPHISM + NEUMORPHISM, React, PARALLAX (solo escritorio),
     animaciones THREE.JS y ANIME.JS al hacer scroll, 100% responsiva.
  8. Ejecutable en LOCAL y EN LÍNEA (Render), igual que til_forwarder.

3. ARQUITECTURA
--------------------------------------------------------------------------------
  [til_tracker (NUEVO, React/Vite)] --GET /maneuvers/monitor/?key=XXX-->
  [maylob_backend (+1 endpoint)] --> [MongoDB: filtro man_moni_key + man_moni_enable]

  - Llave en localStorage['til_moni_key'].
  - Respuestas del endpoint: code '1' = ok con datos | '0' = llave sin
    coincidencias (inválida/cambiada) | '-1' = llave malformada/rechazada.
  - El frontend distingue LLAVE INVÁLIDA (borra la guardada y re-pregunta)
    de ERROR DE RED (conserva la llave y ofrece reintentar).
  - Refresco automático de datos cada 60 s (react-query refetchInterval).
  - Defensa en profundidad: la llave se valida con allowlist EN EL FRONTEND
    (UX) y SE RE-SANITIZA EN EL BACKEND (seguridad real; el front nunca es
    frontera de confianza).

6. SEGURIDAD — MODELO DE AMENAZAS Y DEFENSAS (resumen transversal)
--------------------------------------------------------------------------------
  AMENAZA                              DEFENSA (dónde)
  ---------------------------------    -----------------------------------------
  Inyección de operadores NoSQL        typeof === 'string' + sanitizeId allowlist
  (?key[$ne]=, $gt, $where, regex)     + consulta solo por igualdad literal (B-1)
  Extracción masiva probando llaves    Llaves largas aleatorias + rate limit
  (fuerza bruta / enumeración)         30 req/min (B-5) + respuestas genéricas
  XSS reflejado/almacenado             React escapa por defecto; prohibido
  (payloads en llave o en datos BD)    dangerouslySetInnerHTML (U-4); input con
                                       allowlist en KeyGate (S-4/S-6)
  Link GPS malicioso en BD             isSafeGpsLink: solo http/https; bloquea
  (javascript:, data:)                 javascript:/data:; noopener,noreferrer (U-5)
  Fuga de información en errores       Mensajes genéricos, sin stack traces ni
                                       detalles de Mongo (B-1)
  Fuga de la llave                     Nunca viaja en la respuesta del API
                                       (proyección B-1); solo params URL-encoded
  Escritura desde la app de usuario    La app solo consume UN endpoint GET;
                                       cero formularios (U-3)
  PRINCIPIO RECTOR: el frontend valida por UX; el BACKEND es la única frontera
  de confianza y re-sanitiza TODO. Nada del request toca la consulta Mongo sin
  pasar por sanitizeId, y el filtro es siempre igualdad literal.

7. RIESGOS Y DECISIONES
--------------------------------------------------------------------------------
  R-1  Asignación de llaves: hoy man_moni_key solo se puede poner editando la
       BD. Mitigación: Tarea B-3 (asignarla desde el panel admin). DECIDIR.
       Recomendación: llaves generadas aleatorias >=10 chars (allowlist), no
       nombres adivinables tipo "CLIENTE1".
  R-2  Links GPS de terceros bloquean iframe (X-Frame-Options): por eso el
       canvas es el marco visual y el link abre en pestaña nueva.
  R-3  Una llave "válida" sin maniobras activas es indistinguible de una llave
       inexistente (la llave vive solo en las maniobras). Mensaje neutro:
       "Llave sin maniobras activas o inválida". Alternativa futura: colección
       de llaves propia.
  R-4  backdrop-filter y WebGL en dispositivos viejos: fallbacks definidos
